# 성남시 드론·로봇 배송 거점 대시보드 계산 방식 정리

## 목적

이 대시보드는 성남시 내 H3 격자 단위로 드론·로봇 배송 거점 후보를 평가하고, 선택한 제약 레이어와 POI 조건에 따라 거점과 연결 노드를 다시 계산하는 의사결정 보조 도구다.

핵심 질문은 다음과 같다.

- 어디가 드론·로봇 배송 운영에 적합한가?
- 어디에 수요가 높은가?
- 어떤 POI를 분석 대상으로 볼 때 연결 노드와 거점이 달라지는가?
- 제한 조건을 바꾸면 거점 선정 결과가 어떻게 변하는가?

## 입력 데이터

### H3 격자 데이터

`assets/js/data.js`의 `GRID`는 H3 res.9 단위 분석 셀이다. 각 셀은 위도, 경도, 행정동, 구, 제약 레이어 값을 가진다.

주요 필드:

| 필드 | 의미 |
|---|---|
| `lat`, `lon` | 셀 중심 좌표 |
| `dong`, `gu` | 행정동, 구 |
| `sa` | 공역 제한 점수 |
| `so` | 장애물 점수 |
| `sn` | 소음 민감도 점수 |
| `st` | 지형 경사 점수 |
| `sw` | 기상/바람 점수 |

각 제약 점수는 0에서 1 사이의 값으로 해석한다. 0은 해당 조건에서 운영 부적합, 1은 제약이 거의 없는 상태를 의미한다.

### 드론 수요지수

`assets/js/demand.js`에는 행정동 50개에 대한 수요지수가 들어 있다.

주요 필드:

| 필드 | 의미 |
|---|---|
| `dong` | 행정동명 |
| `demand` | 최종 드론 수요지수 |
| `rank` | 수요 순위 |
| `hr` | 주거/인구 기반 지표 |
| `fp` | 유동인구 기반 지표 |
| `cc` | 소비/카드거래 기반 지표 |
| `od` | 주문/배달 친화 기반 지표 |

현재 고수요 기준값은 다음과 같다.

```js
DEMAND_THR = 0.6
```

즉, `demand >= 0.6`인 행정동에 속한 격자는 기본 분석 대상 노드에 포함된다.

### POI 데이터

`assets/js/poi.js`는 `poi_layers.json`을 JS 모듈 형태로 변환한 데이터다.

현재 포함된 POI 개수:

| POI | 키 | 개수 |
|---|---:|---:|
| 공원 | `park` | 512 |
| 상권 | `commercial` | 108 |
| 의료 | `medical` | 128 |
| 지하철 | `subway` | 129 |
| 학교 | `school` | 81 |

## 계산 수식

### 1. 운영 적합도

선택된 제약 레이어만 곱해서 운영 적합도를 계산한다.

```text
운영 적합도 S(cell)
= 선택된 제약 점수들의 곱
= sa × so × sn × st × sw
```

예를 들어 공역, 장애물, 소음, 지형, 기상이 모두 켜져 있으면 다음과 같다.

```text
S(cell) = sa × so × sn × st × sw
```

선택된 제약 레이어가 없으면 해당 셀의 점수는 0으로 처리한다.

### 2. 기본 우선순위

지도 색상과 기본 셀 우선순위는 운영 적합도와 행정동 수요지수를 곱해서 계산한다.

```text
기본 우선순위 P(cell)
= 운영 적합도 S(cell) × 드론 수요지수 D(dong)
```

여기서 `D(dong)`은 `assets/js/demand.js`의 `demand` 값이다.

### 3. 고수요 분석 대상 노드

기본 분석 대상 노드는 다음 조건을 만족하는 H3 셀이다.

```text
S(cell) > 0
and
D(dong) >= 0.6
```

이 노드들은 거점 선정 알고리즘이 커버해야 하는 기본 대상이다.

### 4. POI 기반 분석 대상 확장

POI 토글을 켜면 해당 POI는 가장 가까운 H3 셀로 매핑된다.

```text
POI target cell = nearest_h3_cell(poi.lat, poi.lon)
```

매핑된 셀이 현재 제약 조건에서 운영 가능한 경우, 분석 대상 노드에 추가된다.

현재 POI 가중치는 다음과 같다.

| POI | 가중치 |
|---|---:|
| 상권 | 1.00 |
| 의료 | 0.95 |
| 지하철 | 0.85 |
| 학교 | 0.75 |
| 공원 | 0.55 |

POI가 포함된 대상 노드의 커버 가중치는 다음처럼 계산한다.

```text
대상 노드 가중치 T(cell)
= S(cell) × (D(dong) + POI_weight_sum(cell))
```

하나의 H3 셀에 여러 POI가 매핑되면 POI 가중치는 합산된다.

### 5. 거점 후보

현재 거점 후보는 별도 시설 목록이 아니라, 운영 가능한 H3 셀 전체에서 생성한다.

```text
거점 후보 = { cell | S(cell) > 0 }
```

각 후보는 자기 주변 500m 안에 있는 분석 대상 노드를 커버할 수 있다.

```text
coverage(candidate)
= { target_cell | distance(candidate, target_cell) <= 500m }
```

### 6. 거점 선정 알고리즘

현재 알고리즘은 그리디 셋 커버 방식이다.

1. 아직 커버되지 않은 대상 노드 집합을 만든다.
2. 각 후보 거점이 새롭게 커버할 수 있는 노드의 가중치 합을 계산한다.
3. 가장 높은 가중치 합을 가진 후보를 선택한다.
4. 선택된 후보가 커버한 노드를 제거한다.
5. 최대 10개 거점 또는 목표 커버리지 90%에 도달할 때까지 반복한다.

후보 점수는 다음과 같다.

```text
candidate_score
= Σ T(target_cell)
for target_cell in newly_covered_targets
```

### 7. 연결선

선정된 거점마다 500m 안의 분석 대상 노드 중 우선순위가 높은 상위 5개 노드에 연결선을 그린다.

```text
연결 노드 = top 5 target cells by T(cell)
```

따라서 POI 토글을 켜면 분석 대상 노드가 바뀌고, 그 결과 연결선과 거점 선정 결과도 함께 바뀐다.

## KPI 의미

| KPI | 의미 |
|---|---|
| 전체 H3 셀 | 전체 분석 격자 수 |
| 적합 셀 수 | 현재 제약 레이어 조합에서 `S(cell) > 0`인 셀 수 |
| 선정 거점 수 | 그리디 알고리즘으로 선택된 후보 거점 수 |
| 고수요 커버리지 | 대상 노드 중 500m 반경 안에 커버된 비율 |
| 평균 우선순위 | 전체 셀의 평균 `S(cell) × D(dong)` |

## 화면 구성

### 지도

지도는 H3 셀을 점으로 표시한다. 색상은 기본 우선순위 `P(cell)`을 기준으로 한다.

점 색상이 높은 값에 가까울수록 다음 조건을 동시에 만족한다.

- 드론 운영 제약이 낮다.
- 행정동 수요지수가 높다.

거점 마커는 현재 조건에서 선택된 후보지이며, 원은 500m 서비스 반경이다.

### 제약 레이어

제약 레이어는 운영 가능성과 점수에 직접 영향을 준다.

- 공역 제한
- 장애물
- 소음 민감
- 지형 경사
- 기상/바람

레이어를 끄면 해당 제약은 점수 곱셈에서 제외된다.

### POI 레이어

POI 레이어는 두 가지 역할을 한다.

- 지도에 해당 POI 위치를 표시한다.
- 선택된 POI를 분석 대상 노드에 추가해 거점과 연결선을 재계산한다.

### 차트

| 차트 | 내용 |
|---|---|
| 행정동 드론 수요지수 순위 | 수요지수 상위 15개 행정동 |
| 시간대별 배송 수요 패턴 | 시간대별 수요 비율 |
| 고수요 동 수요 구성요소 비교 | 상위 수요 행정동의 `hr`, `fp`, `cc`, `od` 비교 |
| 동별 적합 셀 비율 | 행정동별 운영 가능 셀 비율 |
| 배송 수단 비교 분석 | 오토바이와 드론·로봇 배송 특성 비교 |

## 현재 해석 시 주의점

- 수요지수는 행정동 단위이고, H3 셀 단위로 세분화된 실제 주문 데이터는 아니다.
- POI는 가장 가까운 H3 셀로 스냅되므로, 실제 도로 접근성이나 건물 출입 위치와 차이가 있을 수 있다.
- 현재 거점 후보는 실제 물류시설 후보지가 아니라 운영 가능한 H3 셀에서 자동 생성된다.
- 500m 서비스 반경은 직선거리 기준이며, 도보·도로망·로봇 이동 가능 경로를 반영하지 않는다.
- POI 가중치는 현재 정책적 가정값이다. 실측 배송량이나 전문가 평가로 보정할 필요가 있다.

## 발전 방향

### 1. 실제 후보지 기반 거점 선정

현재 후보지는 운영 가능한 H3 셀에서 자동 생성된다. 향후에는 다음과 같은 실제 후보지를 별도 데이터로 관리하는 편이 좋다.

- 공영주차장
- 공공청사
- 물류센터
- 지하철역 출입구
- 공원 관리시설
- 대형 상업시설 배송 집하점

그렇게 하면 결과가 “지도상 좋은 위치”에서 “실제로 설치 가능한 위치”로 바뀐다.

### 2. 도로망/보행망 기반 거리 적용

현재는 500m 직선거리다. 로봇 배송을 고려하려면 보행자 도로, 횡단보도, 경사, 단절 구간을 반영해야 한다.

개선 수식 예:

```text
coverage(candidate)
= { target | network_distance(candidate, target) <= service_radius }
```

### 3. POI 가중치 보정

현재 POI 가중치는 정성적 기준이다. 실제 배송 수요나 정책 우선순위에 따라 보정할 수 있다.

예:

```text
POI_weight = category_weight × activity_intensity × time_weight
```

상권은 점심·저녁 시간대에, 의료는 긴급성 기준으로 더 높은 가중치를 둘 수 있다.

### 4. 시간대별 동적 거점

현재 거점은 정적 계산이다. 시간대별 수요를 반영하면 낮/저녁/야간 거점 조합을 다르게 제안할 수 있다.

```text
T(cell, hour)
= S(cell) × (D(dong) × hourly_weight(hour) + POI_weight_sum(cell, hour))
```

### 5. 민감도 분석

사용자가 가중치를 조절하면서 결과 변화를 비교할 수 있으면 정책 설득력이 높아진다.

추가할 만한 컨트롤:

- 수요지수 가중치
- POI별 가중치
- 서비스 반경
- 최대 거점 수
- 목표 커버리지
- 기상 리스크 가중치

### 6. 결과 저장과 비교

현재는 화면에서 즉시 계산하는 방식이다. 시나리오별 결과를 저장하고 비교하면 의사결정 도구로 더 좋아진다.

예:

- 기본 시나리오
- 의료 POI 우선 시나리오
- 상권 POI 우선 시나리오
- 공역 제한 완화 시나리오
- 악천후 시나리오

### 7. 품질 검증 지표 추가

거점 결과의 안정성을 평가하는 지표를 추가할 수 있다.

- 거점 간 중복 커버율
- 커버되지 않은 고수요 비율
- 평균 접근거리
- 최장 접근거리
- POI 유형별 커버율
- 행정동별 형평성 지표
